## 一、什么是云计算

### 1.1 从 MeowTube 的故事说起

课程用 MeowTube（Cody 的猫咪视频平台）串联全课。早期 MeowTube 运行在一间空闲房间里的单台服务器上。当视频爆火时，他们手忙脚乱地采购和接线新服务器来应对流量——流量回落后，多余的服务器闲置，浪费空间和金钱。远程办公也是噩梦——更新网站、修 Bug、打补丁都依赖笨重的 VPN。

然后他们发现了云计算。

### 1.2 NIST 定义

**NIST（美国国家标准与技术研究院）** 给出了业界广泛接受的定义：

> 云计算是一种模型，提供无处不在的、便捷的、按需的网络访问，访问一个可配置计算资源的共享池（如网络、服务器、存储、应用和服务），这些资源可以快速配置和释放，只需最少的管理工作或服务提供商交互。

这个定义包含：**5 个基本特征**、**3 种服务模型**、**4 种部署模型**。

### 1.3 五个基本特征

| 特征 | 含义 | MeowTube 场景 | 优势 | 风险 |
|------|------|--------------|------|------|
| **按需自助服务** | 不需要审批流程，自己就能启动资源 | Cody 自己就能启动服务器、数据库或容器 | 几分钟内就能部署，不用等几周采购硬件 | 容易忘记关闭资源，账单悄悄增长 |
| **广泛的网络访问** | 通过互联网从任何地方访问 | 团队在任何地方都能工作 | 远程团队和全球用户都能使用 | 没有互联网就没有访问；问题解决速度受限 |
| **资源池化** | 多客户共享提供商的基础设施，虚拟化保持隔离 | MeowTube 不需要拥有硬件 | 享受企业级基础设施，无需自己维护 | 共享环境中配置不当可能带来风险 |
| **快速弹性** | 根据需求自动扩缩容 | 猫咪视频爆火时秒级扩容，冷却后自动缩减 | 水平扩展（加更多机器）比垂直扩展（给一台机器加 CPU/RAM）更灵活 | 如果系统架构不适合弹性伸缩，可能变成成本陷阱 |
| **按量计费** | CPU 周期、存储 GB、带宽都被计量，用多少付多少 | 只为实际使用的资源付费 | 账单透明、用量可追踪 | 需要主动监控，否则容易"月底惊喜" |

### 1.4 垂直扩展 vs 水平扩展（补充）

| | 垂直扩展（Scale Up） | 水平扩展（Scale Out） |
|---|---|---|
| **做法** | 给同一台机器加 CPU/RAM/磁盘 | 增加更多机器分担负载 |
| **上限** | 有硬件上限——一台机器能加的资源有限 | 几乎无上限——理论上可以无限加机器 |
| **容错** | 那台机器挂了，全完了 | 一台挂了，其他机器接管 |
| **云计算偏好** | 云可以做，但不推荐 | ✅ 云的核心优势——弹性水平扩展 |

## 二、服务模型——你租的是什么

### 2.1 IT 技术栈的分层

从底到顶的完整技术栈：

```
┌─────────────────────────┐
│  应用程序 (Applications)  │  ← 用户直接使用的软件
├─────────────────────────┤
│  数据 (Data)             │  ← 应用产生和使用的数据
├─────────────────────────┤
│  运行时 (Runtime)        │  ← 代码运行的环境（如 Python, Java）
├─────────────────────────┤
│  操作系统 (OS)           │  ← 管理硬件和软件的中间层
├─────────────────────────┤
│  虚拟化 (Virtualization) │  ← 将物理资源切分为虚拟资源
├─────────────────────────┤
│  服务器 (Servers)        │  ← 物理或虚拟计算资源
├─────────────────────────┤
│  存储 (Storage)          │  ← 硬盘/SSD
├─────────────────────────┤
│  网络 (Networking)       │  ← 连接一切的管道
└─────────────────────────┘
```

**服务模型的本质就是：这个栈从哪一层开始由云提供商管理。**

### 2.2 四种模式对比

```
          你管理的部分          提供商管理的部分
          
On-Prem:  ████████████████████  (全都是你的)
IaaS:     ██████████            ░░░░░░░░░░  (基础设施交出去)
PaaS:     ████                  ░░░░░░░░░░░░░░░░  (平台也交出去)
SaaS:                           ░░░░░░░░░░░░░░░░░░░░  (全交出去)

█ = 你管理    ░ = 提供商管理
```

| | On-Prem（本地部署） | IaaS | PaaS | SaaS |
|---|---|---|---|---|
| **你管理** | 所有 | OS、应用、数据 | 应用代码、数据 | 基本使用 |
| **提供商管理** | 无 | 服务器、存储、网络、虚拟化 | + OS、运行时 | 全部 |
| **类比** | 自己买房，全自己管 | 毛坯房——结构有了，装修自己来 | 精装房——拎包入住，带自己的东西就行 | 酒店——床铺好了，有客房服务，直接住 |

### 2.3 每种模型的典型服务

| 模型 | AWS 代表 | Azure 代表 | GCP 代表 | 用途 |
|------|---------|-----------|---------|------|
| **IaaS** | EC2 | Virtual Machines | Compute Engine | 虚拟服务器——自己安装 OS 和软件 |
| **PaaS** | Elastic Beanstalk | App Service | App Engine | 上传代码就能跑，不管底层 |
| **SaaS** | — | Microsoft 365 | Google Workspace | 登录即用——邮件、文档、视频会议 |

### 2.4 MeowTube 的选择

| 需求 | 选择 | 模型 | 理由 |
|------|------|------|------|
| 运行视频上传/处理/流媒体平台 | AWS EC2 | IaaS | 需要对底层有完整控制 |
| 快速部署新功能（评论、标签、搜索） | Azure App Service | PaaS | 专注写代码，不折腾环境 |
| 日常办公工具（邮件、文档、视频会议） | Microsoft 365 | SaaS | 零维护，登录就用 |

**现实中的团队几乎不会只用一种模型——而是根据不同需求混合使用。**

## 三、部署模型——你的云跑在哪里

### 3.1 四种部署模型

| 模型 | 本质 | 类比 |
|------|------|------|
| **公有云（Public）** | 共享基础设施，按需使用 | 繁忙的公寓楼——有自己的房间，但共享电梯和管道 |
| **私有云（Private）** | 专属基础设施，只有你使用 | 独栋别墅——没有邻居，完全隐私，但费用和维护都更高 |
| **混合云（Hybrid）** | 公有+私有，通过统一管理面连接 | 同时租别墅（敏感工作）和市中心公寓（日常） |
| **多云（Multi-cloud）** | 使用多个不同供应商的云 | 在不同地方租不同房东的房子 |

### 3.2 各模型详解

#### 公有云（Public Cloud）

```
┌────────────────────────────────────────┐
│           云提供商的数据中心              │
│                                        │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐  │
│  │MeowTube │ │ 公司B   │ │ 公司C   │  │
│  │的 VM    │ │的 VM    │ │的 VM    │  │
│  └─────────┘ └─────────┘ └─────────┘  │
│     虚拟化隔离，共享物理基础设施          │
└────────────────────────────────────────┘
```

- ✅ 弹性伸缩、低维护、全球覆盖、即开即用
- ❌ 控制较少、合规配置需自己做、供应商依赖

#### 私有云（Private Cloud）

- ✅ 完全控制数据位置、强隔离、可自定义安全策略
- ❌ 成本高（不共享）、弹性差、需要更多自行运维

#### 混合云（Hybrid Cloud）

MeowTube 将公共功能（视频流媒体）放在公有云上追求弹性和速度，将敏感数据（用户账户、支付）放在私有环境中追求安全。Azure 提供统一管理界面将两者连接。

- ✅ 兼顾灵活性和安全性，按需分配工作负载
- ❌ 管理两套基础设施的复杂度；数据在环境间流动的安全挑战；成本可能因重复服务而增加

#### 多云（Multi-cloud）

MeowTube 同时使用 AWS 和 Microsoft 的服务——无论是否有意为之，他们已经是多云用户了。

- ✅ 无单点故障、避免供应商锁定、可选择各家最优服务
- ❌ 多套计费系统、多个管理界面、团队需要学习多个平台

## 四、核心云服务——四大构建块

几乎所有云应用都由同样的四个核心服务驱动：

```
┌─────────────────────────────────────────────┐
│                 云应用                        │
│                                             │
│  ┌──────────┐  ┌──────────┐                 │
│  │  计算     │  │  存储     │                 │
│  │ Compute  │  │ Storage  │                 │
│  │ 运行代码  │  │ 保存文件  │                 │
│  └──────────┘  └──────────┘                 │
│                                             │
│  ┌──────────┐  ┌──────────┐                 │
│  │  数据库   │  │  网络     │                 │
│  │ Database │  │ Network  │                 │
│  │ 管理数据  │  │ 连接一切  │                 │
│  └──────────┘  └──────────┘                 │
└─────────────────────────────────────────────┘
```

### 4.1 计算（Compute）——大脑

云计算服务负责执行任务、处理数据、运行决策逻辑。

**三种主要计算模式：**

| 模式 | 特点 | 适合场景 | 各平台名称 |
|------|------|---------|-----------|
| **虚拟机（VM）** | 模拟完整计算机，有独立 OS | 复杂工作负载、需要自定义软件栈 | AWS EC2 / Azure VM / GCP Compute Engine |
| **容器（Container）** | 轻量级，共享 OS 内核，启动快 | 微服务、快速扩缩容 | AWS ECS / Azure AKS / GCP GKE |
| **无服务器（Serverless）** | 只提供代码，基础设施完全由提供商管理 | 事件驱动的短任务 | AWS Lambda / Azure Functions / GCP Cloud Functions |

**Serverless 补充说明：** 名字叫"无服务器"但肯定还是有服务器的——关键区别是你完全看不到也不需要管理服务器。你提供代码，云提供商处理所有运行所需的基础设施。按调用次数和执行时间计费，不用就不收钱。

### 4.2 存储（Storage）——文件柜

虽然我们说"云"，但数据最终还是存储在真实的物理硬件上——遍布全球数据中心的硬盘和 SSD。只不过你不需要看到或管理它们。

**三种云存储类型：**

| 类型 | 工作方式 | 适合场景 | 类比 |
|------|---------|---------|------|
| **块存储（Block）** | 数据切成固定大小的块，无文件结构 | VM 的虚拟硬盘、需要高速读写的活跃数据 | 装在电脑里的 SSD |
| **文件存储（File）** | 传统的文件夹/文件路径结构 | 协作编辑、文档共享、传统应用 | 网络共享驱动器 |
| **对象存储（Object）** | 文件作为"对象"存储，带元数据和唯一 ID，无文件夹层级 | 大型媒体文件、备份、日志 | 带标签的仓库箱子 |

**对象存储是云中最常用的存储类型。** AWS S3、Azure Blob Storage、GCP Cloud Storage 都属于此类。

**存储类/存储层级——按访问频率付费：**

| 层级 | 访问频率 | 成本 | AWS S3 名称 |
|------|---------|------|-----------|
| 标准（Standard） | 频繁访问 | 最高 | S3 Standard |
| 低频访问（IA） | 偶尔访问 | 中等 | S3 Standard-IA |
| 归档（Archive） | 极少访问 | 最低 | S3 Glacier |
| 深度归档 | 几乎不访问 | 极低 | S3 Glacier Deep Archive |

可以设置 **生命周期规则（Lifecycle Rules）** 自动将老数据从高频层移到低频层——比如"30 天后自动转为 Glacier"。

### 4.3 数据库（Database）——工作记忆

每次用户点赞、编辑视频或保存草稿，数据都需要被存储、检索并保持一致。

| 类型 | 适合 | 各平台名称 |
|------|------|-----------|
| **关系型（SQL）** | 结构化数据——用户账户、支付记录 | AWS RDS / Azure SQL / GCP Cloud SQL |
| **非关系型（NoSQL）** | 灵活数据——标签、推荐、编辑历史 | AWS DynamoDB / Azure Cosmos DB / GCP Firestore |

云数据库是 **完全托管的**——提供商处理配置、扩展、备份和复制。你只管数据本身。

### 4.4 网络（Networking）——高速公路

#### 区域与可用区

```
┌──────────────── 区域 (Region): us-east-1 (弗吉尼亚) ─────────────┐
│                                                                   │
│  ┌─────────────┐   ┌─────────────┐   ┌─────────────┐            │
│  │  可用区 1a    │   │  可用区 1b    │   │  可用区 1c    │            │
│  │  独立供电     │   │  独立供电     │   │  独立供电     │            │
│  │  独立冷却     │   │  独立冷却     │   │  独立冷却     │            │
│  │  独立网络     │   │  独立网络     │   │  独立网络     │            │
│  └─────────────┘   └─────────────┘   └─────────────┘            │
│         高速互联                                                  │
└───────────────────────────────────────────────────────────────────┘
```

- **区域（Region）**：地理位置，如 us-east-1（弗吉尼亚）、eu-west-2（伦敦）、ap-south-1（孟买）
- **可用区（Availability Zone, AZ）**：同一区域内独立的数据中心集群，有各自的供电、冷却和网络
- 将工作负载分散到多个 AZ → 即使一个 AZ 挂了，服务仍然在线

#### 负载均衡（Load Balancing）

```
10 个用户同时请求同一个视频

没有负载均衡:                    有负载均衡:
所有请求 → 服务器A               请求 → 负载均衡器 → 服务器A (3个请求)
  (服务器过载，响应慢)                              → 服务器B (3个请求)
                                                   → 服务器C (4个请求)
                                 (均匀分配，全部流畅)
```

负载均衡器还会监控服务器健康——如果某台服务器挂了，自动将流量转到其他健康的服务器。

#### CDN（Content Delivery Network，内容分发网络）

CDN 将热门文件（视频、图片、缩略图）复制到全球各地的 **边缘节点（Edge Location）**，让用户从最近的节点获取内容，大幅减少延迟。

```
没有 CDN:  纽约用户 ──请求──→ 旧金山服务器 ──响应──→ 纽约用户 (延迟高)

有 CDN:    纽约用户 ──请求──→ 纽约边缘节点 ──响应──→ 纽约用户 (极低延迟)
           (文件已缓存在纽约)
```

## 五、安全与成本管理

### 5.1 共享责任模型（Shared Responsibility Model）

**云安全不是提供商一个人的事——而是提供商和用户各负其责。**

```
┌──────────────────────────────────────────────┐
│            用户负责（Security IN the Cloud）    │
│                                              │
│  应用代码安全 / 数据加密 / 访问控制            │
│  IAM 配置 / 安全组配置 / 操作系统补丁          │
├──────────────────────────────────────────────┤
│         提供商负责（Security OF the Cloud）     │
│                                              │
│  物理数据中心 / 服务器硬件 / 网络基础设施       │
│  虚拟化层 / VPC 基础隔离                       │
└──────────────────────────────────────────────┘
```

两个关键安全工具：

| 工具 | 全称 | 作用 | 类比 |
|------|------|------|------|
| **安全组（Security Groups）** | — | 控制网络流量——哪些端口开放、谁能连接 | 大楼的门卫——谁能从哪个入口进来 |
| **IAM** | Identity and Access Management（身份和访问管理） | 控制用户权限——谁能做什么 | 办公室的门禁卡——谁能进哪间屋、打开哪个文件柜 |

**提供商给你工具，但不帮你配置。** 防火墙全开或给所有人管理员权限，提供商不会干预——那是你的责任。

### 5.2 CIA 三元组——安全的三大支柱

| 支柱 | 含义 | 常见威胁 | 防护措施 |
|------|------|---------|---------|
| **机密性（Confidentiality）** | 数据只对授权者可见 | 存储配置错误（公开了不该公开的文件）；代码中硬编码密钥；钓鱼攻击 | 默认私有、使用 Secrets Manager、启用 MFA（多因素认证）、培训员工识别钓鱼 |
| **完整性（Integrity）** | 数据未被篡改 | 未授权的数据修改或删除 | 用 IAM 分离只读和编辑权限、启用日志追溯每一次修改、开启版本控制可回滚 |
| **可用性（Availability）** | 系统在需要时可用 | 服务器故障、区域宕机、流量洪峰 | 跨多个可用区部署、设置负载均衡、配置自动扩缩——但这些都需要你主动配置 |

### 5.3 成本管理

#### CapEx vs OpEx

| | CapEx（资本支出） | OpEx（运营支出） |
|---|---|---|
| **模式** | 大额前期投资购买硬件 | 按月按用量付费 |
| **传统 IT** | ✅ 买服务器、机架、机房 | — |
| **云计算** | — | ✅ 租用计算、存储、网络资源 |
| **特点** | 资产折旧、预算可预测 | 灵活但需要持续监控 |

#### 四大成本优化策略

| 策略 | 问题 | 解决方案 |
|------|------|---------|
| **Right-Sizing（合理调整规模）** | 启动了过大的 VM，95% 时间闲置——像租仓库存一双鞋 | 从小开始，监控使用率，只在性能真正需要时升级。提供商有工具建议更便宜的实例类型 |
| **自动关机** | 开发/测试服务器 24/7 运行，但下班后没人用 | 设置定时任务——晚上关机，早上开机。简单但省钱 |
| **承诺计划** | 稳定工作负载（如 24/7 的生产应用）按需付费太贵 | 签 1 年或 3 年承诺计划，折扣可达 **70%** |
| **标签（Tagging）** | 不知道哪个团队/项目在消耗资源 | 给每个资源打标签（团队、环境、项目），按标签分组查看账单 |

#### FinOps（补充）

**FinOps = 将财务责任引入 DevOps 流程。** 不是一个工具，而是一种文化——让团队对自己消耗的云资源有可见性、有归属感、有优化动力。

## 六、三大云厂商对比

### 6.1 发展简史

| 厂商 | 启动年份 | 背景 | 市场份额 |
|------|---------|------|---------|
| **AWS** | 2006 | Amazon 为自家零售帝国构建了云基础设施，然后发现可以出租给别人 | 全球第一 |
| **Azure** | 2010 | 微软已经统治企业市场，Azure 给这些客户提供直接通往云的通道 | 全球第二 |
| **GCP** | 2011 | Google 将搜索、数据和 AI 方面的经验带入云市场 | 全球第三 |

大多数企业使用不止一家云提供商。**学哪家都行——核心概念是通用的。**

### 6.2 核心服务名称对照表

| 服务类别 | AWS | Azure | GCP |
|---------|-----|-------|-----|
| **虚拟机** | EC2 | Virtual Machines | Compute Engine |

## 七、实操演示

### 7.1 GCP 创建虚拟机

课程演示了在 Google Cloud Platform 上创建 Compute Engine 实例：

**创建步骤：**
1. 进入 GCP Dashboard → Compute Engine
2. 点击 Create Instance
3. 选择机器配置：通用型 e2-medium（2 vCPU，4 GB RAM，约 $25/月）
4. 确认 → 实例出现在列表中，绿色对勾表示运行中

**连接和验证：**
```bash
# 通过 SSH 连接（GCP 提供浏览器内终端）

# 检查磁盘大小
df -h
# 根文件系统约 10 GB——与配置一致

# 检查内存
free -m
# 约 4 GB——与配置一致

# 安装一个终端游戏（纯属娱乐）
sudo apt update && sudo apt install ninvaders -y
```

**清理资源——极其重要：**
1. `exit` 退出 SSH 会话——但 VM 仍在后台运行并计费！
2. 在 Dashboard 中停止实例——不再计算费用，但存储和静态 IP 仍在计费
3. 删除实例——关联的磁盘也一起消失

**核心教训：退出终端 ≠ 停止 VM。停止 VM ≠ 停止所有计费。必须彻底删除才能完全停止计费。**

### 7.2 AWS S3 存储操作

课程演示了在 AWS S3 中创建存储桶并管理文件：

**创建 Bucket（存储桶）：**
1. S3 → Create Bucket
2. 选择通用用途（General Purpose）
3. 名称必须 **全球唯一**
4. 默认设置：
   - Object Ownership = Bucket owner enforced（所有文件归你管理）
   - Public Access = **默认阻止**（安全优先——这与安全课讲的"默认私有"一致）

**可选功能：**
- **版本控制（Versioning）**：保留文件的所有版本，误删/误改可恢复——但注意存储成本增长
- **标签（Tags）**：按项目/部门/数据类型标记，方便管理和计费拆分

**上传与访问：**
- 上传文件时可以选择存储类（Storage Class）
- 直接点击文件 URL → **Access Denied**（因为公共访问被阻止）
- 使用控制台的 Open 按钮 → 以授权身份访问 → 正常显示

**生命周期规则（Lifecycle Rules）：**
```
规则: 30 天后将文件从 Standard 转移到 Glacier Instant Retrieval

效果: 
- 前 30 天: 文件在 Standard 层，快速访问，正常计费
- 30 天后: 自动移到 Glacier，成本大幅降低
- 文件仍可即时访问（Instant Retrieval 的特点）

注意:
- Glacier 有 90 天最低存储期——即使 10 天后删除，仍收 90 天的费
- 转移本身也有费用（不是免费的）
```

**删除：** 必须先删除 Bucket 内的所有对象，然后才能删除 Bucket 本身。

### 核心要点

1. **云计算 = 租用别人的计算机**，但带有按需自助、弹性伸缩、按量计费等关键特征
2. **服务模型决定"你管多少"**：IaaS 给你最多控制和责任，SaaS 给你最少——大多数团队混合使用
3. **部署模型决定"跑在哪里"**：公有追求弹性，私有追求控制，混合兼顾两者，多云避免锁定
4. **四大构建块**（计算/存储/数据库/网络）是所有云应用的基础——不同厂商只是名字不同
5. **安全是共享责任**：提供商管基础设施安全，你管应用和数据安全——工具给你了，配不配是你的事
6. **成本管理需要主动行动**：合理调整规模、自动关机、承诺计划、标签分类——不管就等着月底惊喜
7. **三大厂商（AWS/Azure/GCP）核心概念相通**——学会一家的概念，切换到另一家只是换名字
8. **最重要的一课：创建容易，清理才是真本事**——忘记关闭的资源是最常见的云计算"学费"

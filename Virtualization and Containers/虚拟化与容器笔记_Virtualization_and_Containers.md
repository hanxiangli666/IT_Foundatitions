## 一、为什么需要虚拟化和容器——历史背景与痛点

### 1.1 计算基础设施回顾

在深入虚拟化之前，先明确几个基础概念的递进关系：

```
计算机 (Computer)
  │  一台标准的硬件+OS+应用的栈
  ▼
服务器 (Server)
  │  本质上就是一台计算机，但通常：
  │  - 无显示器、无键盘（Headless，无头模式）
  │  - 7×24 小时运行
  │  - 通过网络响应客户端请求（回忆网络课的 Client-Server 模型）
  ▼
数据中心 (Data Center)
  │  把成千上万台服务器放在一栋安全建筑里
  │  统一供电、制冷、网络
  ▼
云 (Cloud)
  │  将数据中心的计算能力按需出租
  │  用户不需要知道具体在哪台物理机上运行
  └──────
```

每一层的基本架构都是相同的：底层硬件（CPU、内存、存储）→ 中间 OS 和内核 → 顶层应用程序。

### 1.2 传统模式的两大痛点

**痛点一：一台服务器一个应用——硬件严重浪费**

在 1990 年代，企业的标准做法是：每个关键业务服务（邮件、工资系统、备份）各自运行在一台独立的物理服务器上。原因很简单——把多个应用塞在同一台服务器上，它们会争抢资源、互相冲突，一个应用的更新可能导致整台机器崩溃。

结果就是：

- 服务器机房塞满了嗡嗡作响的硬件
- 大部分服务器的 CPU 利用率只有 10-15%——巨大的浪费
- 每增加一个新服务就需要采购新硬件，周期长、成本高

**痛点二："在我的机器上能跑"——环境一致性问题**

开发者面临的经典噩梦：一个应用在自己的电脑上运行正常，部署到测试服务器或生产环境就崩溃了。原因通常是：

- 不同机器上安装了不同版本的运行库
- 操作系统补丁级别不同
- 某个依赖在这台机器上有，另一台上没有

### 1.3 解决方案的演进

这两个痛点催生了两项关键技术：

```
痛点：硬件浪费 + 应用隔离 ──→ 虚拟化（Virtualization）
  │                              将一台物理机拆分成多台虚拟机
  │                              每台 VM 有独立的 OS，强隔离
  │                              但每台 VM 都要跑完整 OS，笨重
  │
  └─── 痛点：VM 太重 + 环境一致性 ──→ 容器化（Containerization）
                                      只打包应用+依赖，共享宿主 OS 内核
                                      极轻、极快、到处都能跑
```

这两项技术并非替代关系，而是互补——现代系统中经常同时使用。

### 1.4 你今天就用到了这两项技术

这些技术不是抽象的理论——你今天大概率已经受益于它们：

- **Zoom / Netflix**：部分服务打包在容器中运行，保持快速和可靠
- **Google Docs / 在线办公**：你的浏览器连接的后端几乎一定运行在数据中心的虚拟机上
- **网上银行 / 云存储**：依赖虚拟化和容器技术来保持在线和弹性扩容

## 二、虚拟机（Virtual Machine）

### 2.1 什么是虚拟机

虚拟化让一台物理计算机同时运行多个相互独立的操作系统。每一个独立运行的系统实例就叫一台 **虚拟机（Virtual Machine, VM）**。

- 你的物理机器叫 **宿主机（Host）**
- 运行在里面的虚拟机叫 **客户机（Guest）**
- 每台 VM 拥有自己的虚拟 CPU、虚拟内存、虚拟硬盘
- 每台 VM 运行自己**完整的操作系统**——可以和宿主机完全不同

**实际场景举例：**

- Cody 用的是 Mac，但客户发来一个 Microsoft Access 文件（一个只有 Windows 才能运行的工具）。她不需要买新电脑或重装系统——只需在 Mac 上启动一台 Windows 虚拟机
- AWS 不会给每个用户分配一台物理服务器——它在同一台物理机上为不同用户创建不同的 VM，各自隔离，各自运行

### 2.2 Hypervisor——虚拟化的核心

让一台物理机同时运行多个 OS 的幕后功臣是 **Hypervisor（虚拟机监控器/虚拟机管理程序）**。它是一层软件（或固件），负责：

- 将物理硬件资源（CPU、内存、存储、网络）**抽象和分割**
- 给每台 VM 分配一份虚拟硬件
- 确保 VM 之间互相隔离

#### 两种类型的 Hypervisor

```
Type 1 — 裸金属型（Bare Metal）          Type 2 — 托管型（Hosted）

┌─────────┐ ┌─────────┐               ┌─────────┐ ┌─────────┐
│  VM 1   │ │  VM 2   │               │  VM 1   │ │  VM 2   │
│(Windows)│ │ (Linux) │               │(Windows)│ │ (Linux) │
├─────────┴─┴─────────┤               ├─────────┴─┴─────────┤
│     Hypervisor      │               │     Hypervisor      │
│  (直接装在硬件上)     │               │   (作为应用安装在     │
│                     │               │    宿主 OS 之上)     │
├─────────────────────┤               ├─────────────────────┤
│      硬件           │               │     宿主 OS          │
└─────────────────────┘               │   (macOS/Windows)   │
                                      ├─────────────────────┤
                                      │      硬件           │
                                      └─────────────────────┘
```

|                    | Type 1（裸金属）                    | Type 2（托管）                                        |
| ------------------ | ----------------------------------- | ----------------------------------------------------- |
| **安装位置** | 直接安装在物理硬件上，没有宿主 OS   | 安装在现有 OS 之上，像普通应用一样                    |
| **性能**     | 更高（没有宿主 OS 的开销）          | 较低（多了宿主 OS 这一层）                            |
| **用途**     | 数据中心、企业生产环境              | 个人电脑、开发测试                                    |
| **代表产品** | VMware ESXi, Microsoft Hyper-V, KVM | VirtualBox, VMware Fusion/Workstation, UTM, Parallels |

### 2.3 VM 的优势

**强隔离（Strong Isolation）**

每台 VM 是完全独立的。一台 VM 崩溃或中毒，其他 VM 和宿主机通常不受影响。这就像商场里的独立店铺——星巴克着火了，隔壁麦当劳还在正常营业（当然前提是消防隔离做得好）。

**运行不同 OS**

可以在一台 Mac 上同时运行 Windows 和 Linux，在一台 Windows 机器上运行 macOS（技术上可行但有授权限制），或者运行已经不再被支持的旧版 OS（如 Windows XP）来使用遗留系统。

**快照与克隆（Snapshots & Cloning）**

- **快照（Snapshot）**：保存 VM 在某一时刻的完整状态。搞砸了？一键回滚到快照点。非常适合做风险操作前的"存档"
- **克隆（Clone）**：复制出一台完全相同的 VM。适合搭建批量的相同环境（如教学实验室的 20 台一模一样的虚拟机）

**数据持久化**

VM 有自己的虚拟硬盘，数据天然持久——就像一个真正的店铺，关门后货物还在里面。重启 VM，数据依然存在。

**网络灵活性**

VM 之间可以像真实计算机一样通过网络通信，常见的虚拟网络模式：

| 模式                          | 特点                                               | 适用场景                       |
| ----------------------------- | -------------------------------------------------- | ------------------------------ |
| **NAT（网络地址转换）** | VM 通过宿主机的网络上网，外部看不到 VM             | 日常上网，简单安全             |
| **桥接（Bridged）**     | VM 获得和宿主机同一网段的 IP，对外就像一台真实设备 | 运行服务器、需要被其他设备访问 |
| **仅主机（Host-Only）** | VM 只能和宿主机通信，不能上网                      | 安全的本地测试环境             |

### 2.4 VM 的局限

| 局限                   | 说明                                                                                 |
| ---------------------- | ------------------------------------------------------------------------------------ |
| **启动慢**       | 每台 VM 要启动一个完整的 OS，通常需要几十秒到几分钟                                  |
| **资源开销大**   | 每台 VM 独占分配的 CPU 和内存。一台 8GB RAM 的电脑，开两个 2GB 的 VM，宿主机只剩 4GB |
| **扩展笨重**     | 想开 10 台新 VM？每台都需要独立安装和配置 OS                                         |
| **GPU 访问受限** | 大多数 VM 无法直接使用宿主机的显卡，不适合游戏、3D 渲染或机器学习训练                |
| **安全非绝对**   | 虽然隔离性强，但 Hypervisor 本身的漏洞（极罕见）理论上可能让恶意软件"逃逸"到宿主机   |

### 2.5 VM 的实际应用场景

| 场景                | 说明                                                                               |
| ------------------- | ---------------------------------------------------------------------------------- |
| **个人/家庭** | 在当前 OS 上试用另一个 OS；安全地测试有风险的软件                                  |
| **企业/办公** | 运行无法升级的遗留系统（如老版本的 ERP、数据库）                                   |
| **云平台**    | AWS EC2、Azure VM、Google Compute Engine 都是在物理服务器上创建 VM，按需出租给用户 |

## 三、容器（Container）

### 3.1 什么是容器

容器是一种将应用及其所有依赖打包在一起、在隔离环境中运行的方式——但**不包含完整的操作系统**。

容器包含的内容：

- 应用代码本身
- 应用依赖的所有软件库和工具
- 配置和设置

容器**不包含**的内容：

- 操作系统内核——容器共享宿主机的 OS 内核

```
虚拟机:                               容器:

┌──────────┐ ┌──────────┐           ┌──────────┐ ┌──────────┐ ┌──────────┐
│  App A   │ │  App B   │           │  App A   │ │  App B   │ │  App C   │
├──────────┤ ├──────────┤           │  + 依赖   │ │  + 依赖  │ │  + 依赖   │
│ 依赖/库   │ │ 依赖/库  │           └────┬─────┘ └────┬─────┘ └────┬─────┘
├──────────┤ ├──────────┤                │            │            │
│ Guest OS │ │ Guest OS │           ┌────┴────────────┴────────────┴────┐
│ (完整OS) │ │ (完整OS) │           │        Container Engine           │
├──────────┴─┴──────────┤           │         (如 Docker)              │
│     Hypervisor        │           ├──────────────────────────────────┤
├───────────────────────┤           │       宿主 OS 内核（共享）        │
│       硬件            │           ├──────────────────────────────────┤
└───────────────────────┘           │           硬件                   │
                                    └──────────────────────────────────┘
每台 VM 背着一个完整 OS               所有容器共享一个 OS 内核
(几百MB ~ 几GB)                      每个容器只带自己需要的东西 (几MB ~ 几百MB)
```

**本质上，容器就是一个"超级加强版的进程"**——它带上了自己运行所需的一切，与其他进程隔离，但不像 VM 那样背着一整个 OS。

### 3.2 镜像（Image）与容器（Container）的关系

这两个概念经常被混用，但它们有本质区别：

|                    | 镜像（Image）                                           | 容器（Container）                          |
| ------------------ | ------------------------------------------------------- | ------------------------------------------ |
| **是什么**   | 一套固定的指令和文件，描述"包含什么、怎么运行"          | 镜像的一个运行实例                         |
| **类比**     | 一张照片——拍完后就不变了                              | 根据这张照片冲洗出来的照片——可以冲很多张 |
| **可变性**   | **不可变（Immutable）**——一旦构建完成就不会改变 | 可变——运行时可以写入文件、修改状态       |
| **数量关系** | 一个镜像可以启动无数个容器                              | 每个容器都是从某个镜像创建的               |

镜像的定义通常写在一个叫 **Dockerfile** 的文件中——这是一个简单的脚本，告诉容器引擎（Container Engine）如何一步步构建镜像。

**Dockerfile 示例（补充）：**

```dockerfile
# 从官方 Python 3.11 镜像开始（基础层）
FROM python:3.11-slim

# 设置工作目录
WORKDIR /app

# 复制依赖文件并安装
COPY requirements.txt .
RUN pip install -r requirements.txt

# 复制应用代码
COPY . .

# 告诉容器启动时运行这个命令
CMD ["python", "app.py"]
```

每一条指令（FROM、COPY、RUN 等）都会创建镜像的一个 **层（Layer）**。层是可以缓存和复用的——如果你只修改了应用代码而没有改依赖，Docker 重建镜像时会跳过 `pip install` 那一步，直接使用缓存，大幅加速构建过程。

### 3.3 Docker——最流行的容器平台

**Docker** 是目前使用最广泛的容器平台。其他替代品包括 Podman 和 containerd，但核心概念相同。

Docker 生态系统的核心组件：

```
┌─────────────────────────────────────────────────────┐
│                    Docker 生态                       │
│                                                     │
│  Dockerfile ──构建──→ Image ──运行──→ Container     │
│     (配方)           (蛋糕模具)       (烤出来的蛋糕) │
│                        │                            │
│                        ▼                            │
│                   Docker Hub                        │
│                (镜像仓库，像"应用商店")               │
│                                                     │
│  Docker Desktop: 本机一键安装的图形化工具             │
│  Docker Engine:  底层的容器运行时                    │
│  Docker Compose: 定义和运行多容器应用的工具           │
└─────────────────────────────────────────────────────┘
```

**Docker Hub** 是一个公开的镜像仓库，类似容器世界的"应用商店"。你可以找到数以万计的现成镜像：Web 服务器（nginx）、数据库（MySQL、PostgreSQL）、编程语言环境（Python、Node.js）等——只需一条命令就能下载并运行。

### 3.4 容器的优势

| 优势               | 说明                                                                              |
| ------------------ | --------------------------------------------------------------------------------- |
| **启动极快** | 没有 OS 要启动，应用可以在秒级启动。课程演示中 VM 启动约 1 分钟，容器几乎瞬间完成 |
| **一致性**   | 在你电脑上能跑的容器，在服务器上也一样跑。彻底告别"在我的机器上能跑"的问题        |
| **可移植**   | 构建一次镜像，到处运行——开发机、测试服务器、生产环境、云平台                    |
| **隔离**     | 每个容器有自己的空间，应用和依赖不会互相冲突。出问题了可以直接删除重建            |
| **资源轻量** | 共享 OS 内核，不重复加载系统。同一台机器上可以运行比 VM 多得多的容器              |
| **易于扩缩** | 需要更多实例？秒级复制。不需要了？秒级销毁                                        |

### 3.5 容器的局限

| 局限                      | 说明                                                                                                                                                           |
| ------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **隔离性弱于 VM**   | 容器共享宿主 OS 内核——内核级别的严重 bug 理论上可能影响所有容器和宿主机                                                                                      |
| **OS 类型必须匹配** | Linux 容器需要 Linux 内核，Windows 容器需要 Windows 内核。不能在 Linux 宿主上直接运行 Windows 容器（Docker Desktop 在 Mac/Windows 上通过轻量 VM 解决这个问题） |
| **设计为单一用途**  | 一个容器最好只运行一个服务/任务，不适合"大而全"的复杂应用                                                                                                      |
| **数据默认不持久**  | 容器删除后，内部数据全部丢失（详见下节）                                                                                                                       |

### 3.6 数据持久化——Volume（卷）

容器被设计为 **临时的（Ephemeral）**——你应该随时可以删除一个容器并重新创建一个。这意味着容器内部文件系统中写入的任何数据都是临时的。

课程中的类比很形象：容器就像厨房操作台——做饭的时候方便使用，但做完就清理干净了。如果你想保留成果（蛋糕、剩菜），你需要放进冰箱或柜子——那就是 **Volume（卷）**。

```
没有 Volume:                          有 Volume:

┌──────────────┐                     ┌──────────────┐    ┌────────────┐
│   Container  │                     │   Container  │    │   Volume   │
│  ┌────────┐  │                     │  ┌────────┐  │    │ (外部存储) │
│  │ 数据   │  │                      │  │ 数据 ──┼──┼───► │ 数据持久   │
│  └────────┘  │                     │  └────────┘  │    │ 保存在这里 │
└──────────────┘                     └──────────────┘    └────────────┘
   容器删除 → 数据丢失                  容器删除 → 数据还在 Volume 里
                                       新容器挂载同一个 Volume → 数据恢复
```

Volume 是独立于容器生命周期的存储区域。即使容器被删除，Volume 中的数据依然安全。这就是数据库、日志、用户上传文件等需要持久化的数据的存储方式。

### 3.7 容器网络

容器不仅需要各自隔离，还需要能互相通信以及与外部世界通信。Docker 等容器引擎通过 **虚拟网络（Virtual Network）** 实现这一点：

- 同一个网络中的容器可以互相发现和通信——就像同一个应用的不同部分之间传递消息
- Docker 通常自动设置默认网络
- **端口映射（Port Mapping）** 让外部世界能访问容器内的服务

端口映射的原理在课程 Demo 中详细演示过：

```
你的浏览器 ──→ localhost:8888 ──→ Docker 端口映射 ──→ 容器内 nginx:80

  宿主机端口 8888                     容器内端口 80
  (你选的，只要没被占用)               (nginx 固定监听的端口)
```

### 3.8 容器编排——管理大量容器

当容器数量从几个增长到几十、几百甚至几千个时，手动管理变得不可能。这时需要 **编排工具（Orchestration Tools）**。

**Kubernetes（K8s）** 是目前最主流的容器编排平台，由 Google 开源。它负责：

- 自动化容器的部署、扩展和管理
- 负载均衡——将流量分配到多个容器实例
- 自动恢复——某个容器挂了，自动重启一个新的
- 滚动更新——零停机地更新服务版本

**Docker Compose** 是一个更轻量的工具，适合在单台机器上定义和运行由多个容器组成的应用。你用一个 YAML 文件描述所有服务及其关系，一条命令全部启动。

**Docker Compose 示例（补充）：**

```yaml
# docker-compose.yml
version: '3'
services:
  web:
    image: nginx
    ports:
      - "8888:80"           # 宿主机 8888 → 容器 80
  
  database:
    image: postgres:15
    environment:
      POSTGRES_PASSWORD: mysecretpassword
    volumes:
      - db-data:/var/lib/postgresql/data   # 数据持久化

volumes:
  db-data:                  # 定义一个命名 Volume
```

一条命令 `docker compose up` 就能同时启动 web 服务器和数据库，自动创建网络让它们互相通信。

## 四、VM vs 容器——全面对比

### 4.1 购物中心类比

课程中用了一个贯穿全课的精彩类比，把你的计算机比作一座购物中心（Shopping Mall）：

```
购物中心 = 你的物理硬件
管理团队 = 操作系统（分配资源、维护秩序）

开店的两种方式:

方式一：独立店铺（VM）                方式二：快闪摊位/Kiosk（Container）
┌──────────────────────┐             ┌──────────────────────┐
│ 星巴克独立门店         │             │ 快闪咖啡摊位          │
│ • 有自己的装修和管理   │             │ • 自带员工、工具和库存 │
│ • 有自己的储物间       │             │ • 使用商场的水电和通道 │
│ • 互不干扰、隔音好     │             │ • 没有自己的储物间     │
│ • 但装修费时费钱       │             │ • 搭建快、拆除也快     │
│ • 占地面积大          │             │ • 占地面积小          │
│ • 不好搬到别的商场     │             │ • 随时可以搬走        │
└──────────────────────┘             └──────────────────────┘
  有自己的完整 OS                       共享宿主 OS 内核
  强隔离、数据持久                       轻量快速、易于扩展
  但笨重、慢、贵                         但隔离弱、数据临时
```

### 4.2 对比总结表

| 维度                 | 虚拟机（VM）                              | 容器（Container）                           |
| -------------------- | ----------------------------------------- | ------------------------------------------- |
| **包含内容**   | 完整 OS + 应用 + 依赖                     | 仅应用 + 依赖（共享宿主 OS 内核）           |
| **大小**       | 几 GB ~ 几十 GB                           | 几 MB ~ 几百 MB                             |
| **启动速度**   | 几十秒到几分钟                            | 秒级甚至亚秒级                              |
| **隔离性**     | 强——每台 VM 有独立 OS                   | 较弱——共享内核，一个内核 bug 可能影响全部 |
| **资源消耗**   | 高——每台 VM 占用独立的 CPU/内存         | 低——共享内核，无重复 OS 开销              |
| **数据持久化** | 默认持久（有虚拟硬盘）                    | 默认临时（需要 Volume 才能持久）            |
| **OS 灵活性**  | 可运行任何 OS（Linux、Windows、macOS 等） | 必须匹配宿主 OS 内核类型                    |
| **可移植性**   | 较差——镜像大、配置多                    | 极强——构建一次，到处运行                  |
| **扩展性**     | 慢——每台都要装 OS                       | 快——秒级复制                              |
| **适合场景**   | 需要不同 OS、强隔离、遗留系统             | 微服务、CI/CD、快速扩缩容                   |

### 4.3 真实场景决策

**场景一：在 Mac 上同时运行 Linux 和 Windows 软件，互相隔离**
→ **VM**。容器不能在不同内核类型上运行不同 OS，只有 VM 能做到在同一台物理机上并排运行完全不同的操作系统。

**场景二：快速启动几十个小型 Web 应用，各自使用不同版本的依赖**
→ **容器**。需要速度、隔离和可重复部署，且不需要特定的 OS。

**场景三：扩展一个有多个服务的云应用，同时需要对宿主系统有完全控制**
→ **两者结合**。VM 提供安全基座和 OS 级别控制，容器在 VM 内部提供快速灵活的服务部署。这种混合模式在云环境中非常普遍。

**核心原则：不是"选哪一个"，而是"哪个适合当前需求"。大多数现代系统两者都用。**

### 4.4 常见误区澄清

**关于 VM 的误区：**

| 说法                      | 真/假 | 澄清                                |
| ------------------------- | ----- | ----------------------------------- |
| "VM 比容器更快更轻"       | ❌ 假 | VM 运行完整 OS，比容器更重更慢      |
| "同一时间只能运行一个 VM" | ❌ 假 | 只要硬件资源够，可以同时运行多个 VM |
| "VM 运行自己的 OS"        | ✅ 真 | 这是 VM 的核心特征                  |
| "VM 提供强隔离"           | ✅ 真 | 每台 VM 有独立的 OS，隔离性强       |

**关于容器的误区：**

| 说法                              | 真/假 | 澄清                                                     |
| --------------------------------- | ----- | -------------------------------------------------------- |
| "容器关闭后数据自动保存"          | ❌ 假 | 默认不保存——需要 Volume 才能持久化                     |
| "容器可以在任何宿主上运行任何 OS" | ❌ 假 | Linux 容器需要 Linux 内核，Windows 容器需要 Windows 内核 |
| "容器共享宿主 OS"                 | ✅ 真 | 这是容器轻量的原因                                       |
| "容器比 VM 更可移植"              | ✅ 真 | 镜像小、依赖自包含、到处能跑                             |

## 五、Docker 实操演示

### 5.1 VM 演示——UTM 运行 Windows XP

课程中使用 UTM（macOS 上的虚拟化工具）演示了 VM 的关键特征：

| 观察项               | 结果                                                                                  |
| -------------------- | ------------------------------------------------------------------------------------- |
| **资源分配**   | 0.5 GB 内存、1.7 GB 磁盘。提醒：不能分配超过宿主机实际拥有的资源                      |
| **启动时间**   | **约 1 分钟**（启动完整 Windows XP）                                            |
| **隔离性**     | VM 完全与宿主隔离——里面做任何事都不影响外面。但 VM 仍连接互联网，所以不是零风险     |
| **数据持久化** | 在扫雷游戏中创建高分记录 → 关闭 VM → 重新打开 → 高分还在。虚拟硬盘的数据跨重启持久 |

常见 VM 工具：UTM（macOS）、VirtualBox（跨平台）、VMware Fusion/Workstation、Parallels（macOS）

### 5.2 Docker 基础命令演示

#### 命令一：运行一个"一次性"容器

```bash
docker run rancher/cowsay "Code Cow"
```

- `docker run`：告诉 Docker 启动一个新容器
- `rancher/cowsay`：镜像名。`rancher` 是 Docker Hub 上的用户名，`cowsay` 是镜像名
- `"Code Cow"`：传给 cowsay 程序的参数——它会打印一只 ASCII 牛说这句话
- 如果本地没有这个镜像，Docker 自动从 Docker Hub 下载

```bash
docker ps -a
```

- `docker ps`：显示容器列表（`ps` = Process Status，但这里只显示容器，不是系统进程）
- `-a`（all）：显示所有容器，包括已经停止的
- 你会看到这个容器的状态是 **Exited**——运行完就退出了。这是典型的容器行为：运行任务，输出结果，退出。整个过程在一瞬间完成

#### 命令二：运行一个持续运行的 Web 服务器

```bash
docker run -d -p 8888:80 nginx
```

逐项解析：

| 参数           | 含义                                                                  |
| -------------- | --------------------------------------------------------------------- |
| `docker run` | 启动新容器                                                            |
| `-d`         | **Detached mode（后台模式）**——容器在后台持续运行，不占用终端 |
| `-p 8888:80` | **端口映射**——宿主机的 8888 端口映射到容器内的 80 端口        |
| `nginx`      | 镜像名——一个轻量级、超流行的 Web 服务器                             |

运行后在浏览器访问 `localhost:8888` 就能看到 nginx 的默认欢迎页——你成功"伸手进入"了容器，看到了它正在提供的服务。

**关于端口号的补充说明（来自课程）：**

- 端口号范围是 0-65535
- 通常选择 1024 以上的四位数（如 8080、8888、3000），避开系统保留端口
- 冒号前面的端口号可以随意换（只要没被占用），冒号后面的必须是应用实际监听的端口
- 课程中的类比：软件端口就像物理端口（USB、HDMI）——你通过一个编号连接到特定的服务

#### 命令三：进入容器内部并测试数据持久化

```bash
docker exec -it <container_id> sh
```

| 参数               | 含义                                                                             |
| ------------------ | -------------------------------------------------------------------------------- |
| `docker exec`    | 在一个运行中的容器内执行命令                                                     |
| `-it`            | `-i`（interactive，交互模式）+ `-t`（TTY，终端）。让你可以通过键盘和屏幕交互 |
| `<container_id>` | 容器 ID——只需输入前几个字符，只要唯一就行                                      |
| `sh`             | 在容器内打开一个 Shell 会话                                                      |

在容器内创建测试文件：

```bash
echo "Code Cow" > /usr/share/nginx/html/test.txt
```

- 在浏览器中访问 `localhost:8888/test.txt` → 能看到 "Code Cow"
- **重启容器** → 文件还在（重启不删除数据）
- **删除容器并重新创建** → 文件消失了！

这就是容器数据持久化的核心真相：

- **重启（restart）**不丢数据——容器还是同一个
- **删除并重建（delete + run new）** 丢数据——这是一个全新的容器

要真正持久化数据，必须使用 Volume：

```bash
docker run -d -p 8888:80 -v my-data:/usr/share/nginx/html nginx
```

`-v my-data:/usr/share/nginx/html` 创建一个名为 `my-data` 的 Volume，并挂载到容器内的指定路径。即使删除容器，数据依然保存在 Volume 中。

### 5.3 Docker 常用命令速查（补充）

| 命令                                         | 作用                        |
| -------------------------------------------- | --------------------------- |
| `docker run <image>`                       | 从镜像创建并启动容器        |
| `docker run -d <image>`                    | 后台运行容器                |

## 六、真实应用——微服务架构

### 6.1 传统架构的问题

课程用一个在线购物网站（exfusioncore.com）作为案例。用户可以浏览商品、加入购物车、下单、支付、接收通知。

**在虚拟化和容器出现之前**，这类网站通常是这样运行的：

- 要么构建为 **一个巨大的单体应用（Monolith）**——所有功能（商品搜索、购物车、支付、通知）全部打包在一起
- 要么把每个功能拆成独立应用，但**每个都运行在自己的物理服务器上**

两种方式的问题：

| 问题                 | 说明                                                         |
| -------------------- | ------------------------------------------------------------ |
| **成本高**     | 每个新功能需要新硬件                                         |
| **扩展慢**     | 促销时流量暴增 10 倍？等新服务器到货、手动部署、祈祷不出问题 |
| **部署风险大** | 更新单体应用的任何一部分，都可能导致整个系统崩溃             |

### 6.2 微服务（Microservices）

现代方案是将应用拆分为 **微服务**——每个主要功能作为一个独立的小型应用运行：

```
传统单体架构:                          微服务架构:

┌──────────────────────────────┐     ┌────────┐ ┌────────┐ ┌────────┐
│       一个巨大的应用           │     │ 商品搜索│ │ 购物车  │ │  支付   │
│                              │     └───┬────┘ └───┬────┘ └───┬────┘
│  商品搜索 + 购物车 + 支付     │         │         │         │
│  + 通知 + 心愿单 + ...       │     ┌───┴────┐ ┌───┴────┐
│                              │     │  通知   │ │ 心愿单  │
│  任何一部分出问题 = 全站崩溃   │     └────────┘ └────────┘
└──────────────────────────────┘   
                                     每个服务独立部署和运行
                                     心愿单挂了？其他服务照常运行
```

微服务的核心优势：**每个服务可以独立构建、独立部署、独立更新、独立扩展**。心愿单功能需要更新？只更新心愿单服务，支付和搜索完全不受影响。

### 6.3 现代部署架构——四层嵌套

微服务、容器、VM 和物理服务器如何组合在一起？

```
┌─────────────────────────────────────────────────────┐
│                   物理服务器 (Server)                 │
│              (或云平台提供的硬件实例)                  │
│                                                     │
│  ┌───────────────────────────────────────────────┐  │
│  │              虚拟机 (VM)                       │  │
│  │        (提供 OS 级隔离和安全策略)               │  │
│  │                                               │  │
│  │  ┌─────────────┐  ┌─────────────┐             │  │
│  │  │  Container  │  │  Container  │             │  │
│  │  │ ┌─────────┐ │  │ ┌─────────┐ │             │  │
│  │  │ │商品搜索  │ │  │ │ 购物车  │ │  ...更多    │  │
│  │  │ │微服务    │ │  │ │ 微服务  │ │  容器       │  │
│  │  │ └─────────┘ │  │ └─────────┘ │             │  │
│  │  └─────────────┘  └─────────────┘             │  │
│  └───────────────────────────────────────────────┘  │
│                                                     │
│  ┌───────────────────────────────────────────────┐  │
│  │              另一台 VM                         │  │
│  │  ┌──────────────┐  ┌──────────────┐           │  │
│  │  │  Container   │  │  Container   │           │  │
│  │  │  支付微服务   │  │  通知微服务   │           │  │
│  │  └──────────────┘  └──────────────┘           │  │
│  └───────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘
```

**从下到上：**

1. **Server（服务器）**：物理机器或云平台实例，提供原始计算资源
2. **VM（虚拟机）**：在服务器上创建隔离环境，提供 OS 级别的安全和控制
3. **Container（容器）**：在 VM 内部运行，打包微服务及其依赖
4. **Microservice（微服务）**：容器内运行的实际业务逻辑

**为什么不跳过 VM 直接在物理服务器上运行容器？** 因为容器共享 OS 内核，隔离性不如 VM。对于处理敏感数据（如支付信息）的服务，VM 的强隔离是必要的安全保障。

### 6.4 弹性扩容——当流量暴增时

假设双十一大促，10000 名用户同时涌入网站。商品搜索服务压力巨大：

```
正常时:                              大促时:
┌──────────┐                        ┌──────────┐ ┌──────────┐ ┌──────────┐
│ 搜索容器  │  × 20 个               │ 搜索容器  │ │ 搜容器   │ │ 搜索容器 │ × 1500 个
└──────────┘                        └──────────┘ └──────────┘ └──────────┘
                                   
用 Kubernetes 自动完成:
1. 检测到流量激增
2. 用现有的搜索服务镜像启动更多容器
3. 将请求分配到所有容器
4. 需要更多计算资源？启动更多 VM，在上面再开容器
5. 流量回落？自动缩减容器数量

整个过程在秒级完成，几乎全自动
```

课程中提到的真实案例：黑色星期五期间，Amazon 可能将单个微服务（如搜索自动补全）从 20 个容器扩展到超过 1500 个。

### 6.5 零停机更新与故障隔离

**更新：**

1. 构建一个包含修复的新容器镜像
2. 测试
3. 用 Kubernetes 执行 **滚动更新（Rolling Update）**——逐步用新容器替换旧容器
4. 旧容器被优雅地关闭
5. 全程没有停机，用户无感知

**故障隔离：**

- 如果一个新的"优惠券"微服务崩溃了，只有它自己的容器受影响
- 商品搜索、支付、购物车继续正常运行
- Kubernetes 自动尝试重启崩溃的容器

这就是微服务 + 容器 + VM 组合的威力：**速度 + 安全 + 弹性**。

## 七、底层隔离技术原理（补充）

课程中提到容器"共享 OS 内核但互相隔离"，但没有解释这种隔离是如何实现的。这里做一个重要补充。

### 7.1 Linux 容器的隔离基石

容器的隔离不是靠虚拟化硬件，而是靠 Linux 内核的两项原生机制：

#### Namespaces（命名空间）——"看到什么"

Namespace 让每个容器拥有自己的独立视图，看不到其他容器和宿主机的资源：

| Namespace      | 隔离的内容                                          |
| -------------- | --------------------------------------------------- |
| **PID**  | 进程 ID——容器内的进程只能看到自己容器里的进程     |
| **NET**  | 网络——每个容器有自己的网络接口、IP 地址、端口空间 |
| **MNT**  | 文件系统挂载点——每个容器有自己的文件系统视图      |
| **UTS**  | 主机名——每个容器可以有自己的主机名                |
| **IPC**  | 进程间通信——容器间不能直接通过共享内存通信        |
| **USER** | 用户和用户组——容器内的 root 不等于宿主机的 root   |

#### cgroups（控制组）——"能用多少"

cgroups 限制每个容器能使用的资源量：

| 资源               | 控制                                               |
| ------------------ | -------------------------------------------------- |
| **CPU**      | 限制容器最多使用多少 CPU 时间                      |
| **内存**     | 限制容器最多使用多少 RAM，超出就被 OOM Killer 终止 |
| **磁盘 I/O** | 限制读写速度                                       |
| **网络**     | 限制带宽                                           |

这两者结合，就是容器隔离的全部秘密：**Namespace 控制"能看到什么"，cgroups 控制"能用多少"**。

这也解释了为什么容器的隔离弱于 VM——VM 通过 Hypervisor 虚拟了整套硬件，每台 VM 运行独立的 OS 内核；而容器只是通过内核特性做进程级隔离，共享同一个内核。

### 核心要点回顾

1. **虚拟化解决了"一机一应用"的浪费**——Hypervisor 让一台物理机分裂成多台独立的虚拟机，各自运行完整 OS
2. **容器解决了"VM 太重"和"环境不一致"**——共享宿主 OS 内核，只打包应用+依赖，秒级启动，到处能跑
3. **VM 的优势是强隔离和 OS 灵活性**——需要不同 OS 或安全边界时用 VM
4. **容器的优势是轻量、快速、可移植**——需要快速部署和弹性扩缩时用容器
5. **数据持久化是容器的"陷阱"**——容器默认临时，必须用 Volume 才能保存数据
6. **现实世界通常两者都用**——VM 提供安全基座，容器在 VM 内提供快速执行
7. **微服务 + 容器 + VM + 云 = 现代应用架构**——从小型创业公司到 Amazon 级别的全球系统，都用这套组合
8. **容器隔离靠 Namespace（看到什么）+ cgroups（用多少）**，不如 VM 的硬件级虚拟化强，但足够应付绝大多数场景
